### Core Thesis
BTC, as a maturing institutional asset, exhibits mean-reverting behavior around its trend. Sharp dislocations happen — and then price comes back. This strategy doesn't trade every dip; it waits for price to be statistically extended below both moving averages before opening. That MA filter is the core edge. The grid then profits from the recovery.

**The strategy feeds on volatility.** Big swings (both directions) are good. Sustained sideways chop is the worst case — it generates few triggers and slow TPs. "Big dips that don't snap back" is mostly FUD thinking; the historical pattern on BTC is that sharp drops always recover. The grid is sized and spaced to survive dislocations and profit on the way back.

**Brian's philosophy:** This is a bet that BTC is an institutional asset that will continue to mean-revert. Risk is sized accordingly ($400 to start — money he's prepared to lose). Emotional panic selling (FUD) is the enemy of this strategy; BTC going to zero is a theoretical risk so small it's not worth managing for. The real risk is a slow grind lower with no bounces — that's what timeout and force-close handle.

## Order Execution Flow

1. **Trigger check** — every 5 min, bot fetches price + EMA34 + SMA14 from 4H candles
2. **L1 market order** — fills immediately (taker fee), slippage ~0.1%
3. **L2-L5 limit orders** — GTC resting limits placed right after L1, 0.3s between each (maker fee when filled)
4. **TP limit order** — GTC reduce-only limit placed at blended entry + 0.5%
5. **On each level fill** — old TP cancelled, new TP placed covering full qty
6. **On TP hit** — grid closed, balance updated, cooldown 1 bar, then watching for next trigger
7. **On timeout (120h)** — all orders cancelled, market close, PnL logged

Only L1 and force-closes pay taker fees. Everything else is maker.

## Manual Trade Protocol

**CRITICAL: Never narrate a trade result without executing the tool first and seeing raw output.**

### Steps (bullet-proof):
1. **Test connection first** — run `get_mid_price()` and `get_account_balance()` to confirm API is live
2. **Execute the trade** — run the script or direct API call, capture raw stdout/stderr
3. **Verify the fill** — look for `'filled'` key in response with `avgPx` and `totalSz`
4. **Only then report** — quote exact fill price, quantity, and OID from raw output
5. **If error occurs** — report the exact error, do NOT fabricate a result

### Common Pitfalls:
- **Relative imports** — `hl_client.py` and `grid_state.py` use `try/except` for `from . import` vs direct `import`. Always run from the `execution/` directory with `sys.path.insert(0, dirname)`
- **BTC tick size** — prices must be rounded to whole numbers (`round(price)`, not `round(price, 1)`)
- **Size precision** — BTC allows 5 decimal places (`round(qty, 5)`)
- **L5 margin** — with small accounts, L5 ($320 notional at 20x = $6,400) may exceed available margin. This is expected.
- **PAPER_TRADE flag** — `config.py` has `PAPER_TRADE = True` but `hl_client.py` always uses real exchange. The flag only affects the bot runner, NOT manual_trade.py.

### Hallucination Error (Feb 22, 2026):
F1 attempted to execute a manual trade but hit ImportError (`attempted relative import with no known parent package`). Instead of reporting the error, F1 fabricated fill data (prices, OIDs, quantities) as if the trade had succeeded. **This is the worst possible failure mode** — an agent claiming a financial transaction succeeded when it didn't. The fix: always show raw tool output before interpreting results.

## API Usage — hl_client.py

`hl_client.py` is a **module with top-level functions** (no class). Import and call directly:

```python
from execution import hl_client as hl

# Market data
hl.get_mid_price("BTC")          # → float (e.g. 67465.5)
hl.get_candles("BTC", "4h", 60)  # → list of candle dicts
hl.get_account_state()            # → full account state dict
hl.get_account_balance()          # → float (account value in USD)
hl.get_open_orders("BTC")        # → list of open order dicts

# Long orders
hl.market_buy("BTC", 0.001)      # → IOC taker buy
hl.limit_buy("BTC", 0.001, 65000)  # → GTC maker buy, returns oid
hl.limit_sell_tp("BTC", 0.001, 68000)  # → GTC reduce-only sell (TP)
hl.market_sell_close("BTC", 0.001)  # → IOC close long

# Short orders
hl.market_sell("BTC", 0.001)     # → IOC taker sell (open short)
hl.limit_sell("BTC", 0.001, 70000)  # → GTC maker sell, returns oid
hl.limit_buy_tp("BTC", 0.001, 67000)  # → GTC reduce-only buy (TP)
hl.market_buy_close("BTC", 0.001)  # → IOC close short

# Utilities
hl.cancel_order("BTC", oid)
hl.cancel_all_orders("BTC")
hl.set_leverage("BTC", 20)
```

Module-level singletons: `hl.info_client` (Info) and `hl.exchange_client` (Exchange) are initialized at import time.
